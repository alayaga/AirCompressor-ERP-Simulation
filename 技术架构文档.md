# 空压机虚仿平台 — 技术架构文档

## 1. 架构概览

### 1.1 项目定位

本平台是一个 **Unity 2022.3 URP** 驱动的 **ERP 业务仿真培训系统**，以空压机制造企业为蓝本，通过第一人称 3D 交互模拟从客户询单到产品交付的全链路业务流程。平台采用 **协程驱动的流程引擎** 作为核心编排层，以 **ScriptableObject** 作为配置中心，实现了"流程定义零代码、单据配置可视化"的架构目标。

### 1.2 技术栈

| 层面 | 技术选择 |
|------|---------|
| 运行时 | Unity 2022.3.62f3c1, C# 9.0, .NET Standard 2.1 |
| 渲染 | URP（Universal Render Pipeline），三档画质配置 |
| UI | Unity UI (uGUI) + TextMesh Pro |
| 数据配置 | ScriptableObject（BillData, DialogueData, FlowConfig） |
| 输入 | Unity Input System (Legacy Input Manager) |
| 架构模式 | 单例模式 + 协程状态机 + Component Pattern + 反射 |

## 3. 核心架构模式

### 3.1 单例管理器矩阵

项目中大量使用 **MonoBehaviour Singleton** 模式，每个管理器独立承担一个横切关注点。所有单例均实现了 `DontDestroyOnLoad` 以支持跨场景持久化。

```
                    ┌─────────────────┐
                    │   FlowManager   │  ← 流程注册/调度/队列
                    └────────┬────────┘
                             │ 协程托管
                    ┌────────▼────────┐
                    │ FlowTaskInteg.. │  ← 流程→UI 桥接
                    └────────┬────────┘
                             │
    ┌───────────────────────┼───────────────────────┐
    │                       │                       │
┌───▼──────┐  ┌────────────▼──┐  ┌─────────────────▼──┐
│UIManager │  │InteractionMgr │  │AnimationService    │
│(面板显隐)│  │(E键交互调度)   │  │(车间+质检动画)      │
└──────────┘  └───────────────┘  └────────────────────┘
    │                       │
┌───▼──────┐  ┌────────────▼──┐  ┌─────────────────────┐
│ObjectMgr │  │DemandManager  │  │PositionManager      │
│(物体查找)│  │(需求生成)      │  │(位置瞬移)            │
└──────────┘  └───────────────┘  └─────────────────────┘
                    │
            ┌───────▼───────┐
            │InventoryMgr   │
            │(11种零件库存)  │
            └───────────────┘
```

**设计考量**:
- 每个 Manager 职责单一，通过 Inspector 暴露配置，运行时通过 `Instance` 静态属性访问
- 避免了 Unity 场景中 `GameObject.Find` 的运行时开销（改为 Inspector 预绑定）
- 缺点：单例间存在隐式依赖，单元测试较困难；初始化顺序依赖 Unity 的 `Awake` 调用时机

#### 协程安全调度: FlowRunner

`FlowRunner` 解决了 Unity 协程的核心痛点——`StopCoroutine` 无法保证立即生效：

```
public IEnumerator RunSafe(IEnumerator routine)
{
    _isRunning = true;
    while (_isRunning && routine.MoveNext())  // ← 每次迭代检查标志位
    {
        var current = routine.Current;
        if (current is IEnumerator nested)
            yield return RunSafe(nested);     // ← 递归展开嵌套协程
        else
            yield return current;
    }
    _isRunning = false;
}

public void Stop() => _isRunning = false;    // ← 安全的"软中断"
```

### 3.4 交互系统的校验链

```
玩家按下E键
     │
     ▼
InteractionManager.HandleInteractionInput()
     │
     ▼
TryInteract()
     │
     ├─→ PlayerInteractionDetector.currentInteractable == null?
     │     └─ YES → 忽略
     │
     ├─→ IsValidTargetForCurrentStep(target)?
     │     │
     │     ├─→ FlowTaskIntegration.Instance.GetCurrentFlow()
     │     ├─→ 通过反射获取当前步骤的 targetNPC
     │     ├─→ 比较 target.npcName vs step.targetNPC
     │     └─→ 不匹配 → 忽略
     │
     ├─→ target 上是否有 WorkshopAnimationTrigger?
     │     └─ YES → ShouldTriggerAnimation()?
     │           └─ YES → TriggerAnimation(callback) → callback中调用 target.OnInteract()
     │
     ├─→ target 上是否有 QualityInspectionTrigger?
     │     └─ YES → ShouldTriggerInspection()?
     │           └─ YES → TriggerInspection(callback) → callback中调用 target.OnInteract()
     │
     └─→ target.OnInteract()
           │
           └─→ FlowStepTracker.CompleteStep()
                 └─→ FlowTaskIntegration.CompleteCurrentStep()
                       └─→ currentFlow.MarkStepComplete()
```

**设计亮点**: 动画触发器的 **拦截器模式**——交互流程优先检查动画条件，满足时先播动画再回调，不满足时直接走交互逻辑。这种设计使得车间动画和质检动画对流程步骤完全透明。

### 3.5 分支流程架构

`StandardSalesFlow` 和 `CustomSalesFlow` 实现了 **主流程 → 分支子流程** 的嵌套编排模式。

```
┌──────────────────────────────────────────────────┐
│              StandardSalesFlow (主流程)            │
│                                                    │
│  步骤1: 分支选择 (销售-PMC)                        │
│  步骤2: 分支选择 (销售-交付)       ← 5 个分支选择点 │
│  步骤3: 分支选择 (生产)                            │
│  步骤4: 分支选择 (采购)                            │
│  步骤5: 分支选择 (生产部采购)                       │
│  步骤6: 继续主流程后续步骤                          │
│                                                    │
│  分支调度: SelectBranch(int index)                │
│    → _currentBranchFlow = new SubFlow()           │
│    → yield return StartBranchFlow<SubFlow>()       │
│    → 子流程结束后, _currentBranchFlow = null       │
│    → 回到分支选择循环                               │
└────────────────────┬─────────────────────────────┘
                     │
      ┌──────────────┼──────────────┬──────────────┐
      │              │              │              │
┌─────▼──────┐ ┌────▼──────┐ ┌─────▼─────┐ ┌─────▼──────┐
│SalesBranch │ │Delivery   │ │Production │ │Purchase    │
│Flow (197行)│ │Flow (348) │ │Flow (514) │ │Flow (243)  │
│            │ │           │ │           │ │            │
│ 5 步骤     │ │ 14 步骤   │ │ ~40 步骤  │ │ 12 步骤    │
│            │ │ + 自动生产 │ │           │ │            │
└────────────┘ └───────────┘ └───────────┘ └────────────┘
```

**分支流程的关键技术细节**:

1. **反射透传**: `InteractionManager` 和 `FlowTaskIntegration` 通过反射读取 `_currentBranchFlow` 私有字段，确保交互校验和步骤完成通知能路由到正确的子流程
2. **自动触发**: `StandardDeliveryFlow` 在库存不足时自动触发 `StandardProductionFlow` 子流程
3. **可重复进入**: 分支选择点循环等待，允许玩家多次进入同一分支

## 4. 关键设计决策与权衡

### 4.1 为什么用协程而不是 async/await？

| 维度 | 协程 (Coroutine) | async/await (Task) |
|------|-----------------|-------------------|
| **与 Unity 主线程集成** | ✅ 原生支持，每帧驱动 | ❌ 需要 SynchronizationContext |
| **帧级等待操作** | ✅ `WaitForSeconds`, `WaitUntil`, `WaitForFixedUpdate` | ❌ 需自行实现 |
| **GameObject 生命周期安全** | ✅ StopCoroutine 直接终止 | ⚠️ 需要 CancellationToken |
| **Unity API 调用** | ✅ 安全 | ❌ 必须回到主线程 |
| **异常处理** | ⚠️ try-catch 可行但不优雅 | ✅ 原生支持 |

**结论**: 对于帧级用户交互等待和 Game Object 强依赖的场景，协程是 Unity 项目的最佳选择。但协程的嵌套和停止行为有时不够优雅，因此引入了 `FlowRunner` 作为"安全停止"的补充层。

### 4.2 为什么用反射获取分支流程而不是接口？

`InteractionManager` 和 `FlowTaskIntegration` 通过反射读取 `StandardSalesFlow._currentBranchFlow`：

```csharp
// 当前实现
FieldInfo branchFlowField = typeof(StandardSalesFlow)
    .GetField("_currentBranchFlow", BindingFlags.NonPublic | BindingFlags.Instance);
FlowBase branchFlow = branchFlowField?.GetValue(mainFlow) as FlowBase;
```

**利**: 分支流程类型各异（SalesBranch/Production/Purchase/Delivery），避免为每个 Flow 类定义统一的分支接口
**弊**: 性能开销（但对话/UI 交互场景下纳秒级可忽略）、重构脆弱性、字段名硬编码

**替代方案**: 可以在 `FlowBase` 基类上添加虚属性 `virtual FlowBase CurrentBranchFlow => null`，各子类重写。这是后续优化的推荐方向。

### 4.3 为什么用 ScriptableObject 而不是 JSON/XML 做配置？

| 维度 | ScriptableObject | JSON/XML |
|------|-----------------|----------|
| **Unity Editor 可视化编辑** | ✅ Inspector 直接编辑 | ❌ 需要外部工具 |
| **引用 Unity 资源** | ✅ 可直接引用 Sprite/Prefab | ❌ 只能存路径字符串 |
| **热重载安全** | ✅ Play 模式下不丢失 | ⚠️ 需手动处理 |
| **包体 / 加载性能** | ✅ 编译时序列化为二进制 | ❌ 运行时解析 |
| **版本控制友好** | ⚠️ .asset 文件大、merge 难 | ✅ 纯文本 diff |

**结论**: 对于需要引用 Unity 资源的配置（如单据附件 Sprite），ScriptableObject 是唯一正确的选择。项目中的 `BillData`、`DialogueData` 均采用此方案。

### 4.4 为什么交互 NPC 校验放在 InteractionManager 而非各 Flow 内？

**中心化校验** (当前设计):
- `InteractionManager` 持有完整的校验逻辑，通过反射遍历所有 Flow 类型
- 优点: 对所有流程统一的"按错人"提示，不需要每个 Flow 子类重复校验
- 缺点: `GetTargetNPCFromFlow()` 是一个巨大的 `if-else if` 链（160 行），每次新增 Flow 类型必须在此处追加分支

**分布式校验** (替代设计):
- 每个 `FlowBase` 子类暴露 `GetCurrentTargetNPC(): string` 虚方法
- 优点: 符合开闭原则，新增 Flow 无需修改 InteractionManager
- 缺点: 调用链更长，调试时需要追踪到具体 Flow 子类

**推荐**: 在 `FlowBase` 基类上添加 `public virtual string GetCurrentTargetNPC() => null`，各子类重写后，`InteractionManager` 的 `GetTargetNPCFromFlow` 可简化为一行 `return flow.GetCurrentTargetNPC()`。

## 9. 性能与优化策略

### 9.1 已实施的优化

| 策略 | 实现位置 | 说明 |
|------|---------|------|
| **BillData 缓存** | `BillDataLoader._cache` | 首次 `Resources.LoadAll` 后缓存，避免重复 IO |
| **反射缓存** | `InteractionManager` / `FlowTaskIntegration` | `GetField()` 每次调用返回的是相同的 `FieldInfo` 引用（.NET 内部缓存） |
| **协程分帧** | `FlowCoroutine()` 中 `yield return null` | 每个步骤间至少等一帧，避免单帧卡死 |
| **Inspector 预绑定** | 所有 Manager 的引用字段 | 消除 `GameObject.Find` 的 O(n) 查找 |
| **文本缓存** | `InteractionManager.INTERACT_PROMPT_TEXT` | 常量字符串避免 GC 分配 |

### 9.2 潜在优化点

| 问题 | 位置 | 建议 |
|------|------|------|
| **巨大 if-else 链** | `InteractionManager.GetTargetNPCFromFlow()` (160行) | 改为 `FlowBase.GetCurrentTargetNPC()` 虚方法 |
| **重复反射** | `InteractionManager` 和 `FlowTaskIntegration` 各自反射 | 抽象为公共方法或改为虚属性 |
| **步骤数据硬编码** | 各 Flow 的 `InitializeSteps()` 方法 | 可将步骤数据迁移到 ScriptableObject 或 JSON 配置 |
| **BillView 过重** | `BillView.cs` (819行) | 可拆分为 ButtonManager、DataManager、AnimationManager 等子组件 |

## 10. 安全性设计

### 10.1 输入锁定分层

系统在以下场景中分层禁用玩家输入，确保不会出现"对话时还能移动"之类的问题：

```
情况                       锁定范围
─────────────────────────────────────────────
打开单据面板                准心隐藏 + 光标释放 + 玩家移动禁用
对话播放中                  玩家移动禁用 + 交互禁用 (E键屏蔽)
车间动画播放中              交互禁用 (E键屏蔽)
质检动画播放中              交互禁用 (E键屏蔽)
新手引导面板                玩家移动禁用 + 交互禁用
分支选择面板                玩家移动禁用
```

### 10.2 空值防护

所有关键路径均包含空值检查：

```csharp
// FlowBase.WaitForBillComplete — 5 层防护
var billData = BillDataLoader.GetConfig(billType);
if (billData == null) { yield break; }        // 1. 配置缺失

UIManager.Instance.ShowUI(billType);
if (!UIManager.Instance.TryGetUI(billType, out GameObject panel) || panel == null)
    { yield break; }                           // 2. 面板未注册

var billView = panel.GetComponent<BillView>();
if (billView == null) { yield break; }         // 3. 组件缺失
```

### 10.3 错误日志级别

| 级别 | 使用场景 | 示例 |
|------|---------|------|
| `Debug.LogError` | 阻塞性错误（配置缺失、组件缺失） | `"[FlowBase] FAIL: 未找到 BillData"` |
| `Debug.LogWarning` | 非阻塞异常（默认值回退） | `"[BillData] 未找到角色配置，使用默认空列表"` |
| `Debug.Log` | 正常流程状态追踪 | `"[FlowBase] OK: 配置已加载"` |
